# نظام إدارة مكاتب الاستقدام — الخلفية (Backend)

خلفية حقيقية متعددة المكاتب (Multi-tenant) مع نظام صلاحيات (RBAC):
REST API + قاعدة بيانات MySQL + مصادقة JWT + لوحات إدارة + بوابة عميل + خدمة إشعارات قابلة للربط بمزوّد فعلي. بدون ربط مع مساند — الموظف يرفق المستندات ويتابع المراحل.

## المتطلبات
Node.js 18+ مع قاعدة بيانات MySQL (عبر mysql2).

## التشغيل
```bash
cp .env.example .env
npm install
npm start
```
- API: `http://localhost:4000`
- لوحة مدير المنصة: `/admin.html`
- لوحة إدارة مستخدمي المكتب (للمشرف): `/users.html`
- إدارة العمالة المتوفرة: `/workers.html`
- كتالوج العميل العام: `/catalog.html?key=...`
- بوابة المتابعة: `/track.html`

تُنشأ قاعدة `rms.db` تلقائياً مع بيانات تجريبية عند أول تشغيل.

## هيكل الصلاحيات
| الدور | الوصف | الصلاحيات |
|---|---|---|
| **مدير المنصة** (super_admin) | يدير المنصة كلها | إنشاء المكاتب وحسابات مشرفيها، إيقاف/تفعيل المكاتب. لا يصل لبيانات المكاتب التشغيلية. |
| **مشرف المكتب** (supervisor) | كل الصلاحيات داخل مكتبه | كل ما يخص الطلبات والمالية والمستندات واللوحة + **إدارة مستخدمي مكتبه**. |
| **محاسب** (accountant) | الجانب المالي | عرض الطلبات، عرض/تعديل المالية، المستندات، لوحة التحكم. |
| **موظف متابعة** (follow_up) | تشغيل الطلبات | عرض/تعديل الطلب، نقل المراحل، المستندات، إدارة العمالة، لوحة التحكم. |
| **موظف استقبال** (reception) | استقبال العملاء | عرض/إنشاء/تعديل الطلب والعميل، المستندات، إدارة العمالة. |

**عزل البيانات:** كل مكتب يرى ويعدّل طلباته ومستخدميه فقط؛ لا تسرّب بين المكاتب (مطبّق على مستوى قاعدة البيانات بناءً على المكتب المرتبط بالتوكن).

## الواجهات البرمجية
**المصادقة**
- `POST /api/auth/login` — يرجّع `{ token, user{role,permissions,officeId,...} }`
- `GET /api/me` — بيانات المستخدم الحالي وصلاحياته

**مدير المنصة** (super_admin)
- `GET /api/admin/offices` — قائمة المكاتب + إحصاءات
- `POST /api/admin/offices` — إنشاء مكتب + مشرفه `{officeName, supervisor{name,username,password}}`
- `PATCH /api/admin/offices/:id` — `{active}` تفعيل/إيقاف

**إدارة مستخدمي المكتب** (صلاحية users.manage)
- `GET /api/users/roles` — الأدوار وصلاحياتها
- `GET /api/users` — مستخدمو مكتبك
- `POST /api/users` — `{name,username,password,role}`
- `PATCH /api/users/:id` — `{active}` أو `{role}`

**بيانات المكتب** (معزولة حسب المكتب، محمية بالصلاحيات)
- `GET /api/orders` (orders.view) · `POST /api/orders` (orders.create)
- `GET /api/orders/:id` (orders.view) · `PATCH /api/orders/:id` (orders.edit و/أو finance.edit مع تصفية الحقول)
- `POST /api/orders/:id/advance` (orders.advance) — ينقل المرحلة ويرسل إشعاراً
- `POST /api/orders/:id/documents` (documents.manage) · `GET /api/orders/:id/documents/:docId`
- `GET /api/dashboard/summary` (dashboard.view)

**عام**
- `POST /api/public/track` — متابعة العميل برقم الطلب + آخر 4 أرقام جوال

### قواعد نقل المراحل (تُطبّق آلياً)
- «تعاقد» يتطلب إرفاق العقد · «تأشيرة» يتطلب إرفاق التأشيرة + رقمها · «التسليم» يتطلب فاتورة السداد.
- عند الوصول يُحتسب الضمان (90 يوماً)، وكل انتقال يُسجَّل ويُرسل إشعار للعميل.

## تفعيل الإشعارات الحقيقية
في `.env` غيّر `NOTIFY_PROVIDER`:
- **twilio**: `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_FROM=whatsapp:+...`, `TWILIO_CHANNEL=whatsapp`
- **unifonic**: `UNIFONIC_APPSID`, `UNIFONIC_SENDER`
- **console** (افتراضي): يطبع الرسائل في الطرفية للتجربة. تُحوَّل الأرقام تلقائياً للصيغة الدولية.

**العمالة المتوفرة + كتالوج العميل** (جديد)
- `GET /api/workers` (workers.view) — قائمة العمالة (داخلي: يشمل الجوال ورقم الجواز)
- `POST /api/workers` (workers.manage) — إضافة عاملة (multipart: الحقول + `photo` + `cv`)
- `PATCH /api/workers/:id` (workers.manage) — تعديل البيانات/الحالة أو استبدال الملفات
- `GET /api/workers/:id/photo` · `GET /api/workers/:id/cv` (workers.view) — تنزيل داخلي
- `GET /api/workers/catalog-link` — رابط بوابة الكتالوج الخاص بالمكتب
- **عام:** `GET /api/public/catalog?key=KEY` — الجنسيات والعمالة المتوفرة (بيانات مختصرة فقط، بدون جوال أو رقم جواز)
- **عام:** `GET /api/public/catalog/photo|cv?key=KEY&id=ID` — صورة/سيرة العاملة
- **عام:** `POST /api/public/catalog/reserve` — حجز عاملة من صفحة السيرة (يطلب `phone`)؛ يحوّل حالتها إلى «محجوزة» ويُسجّل رقم العميل
- `GET /api/workers/reservations` (workers.view) — طلبات الحجز الواردة من العملاء (الاسم + جوال العميل + التاريخ)

> حالات العاملة: متوفرة / محجوزة / تم التعاقد — ولا تظهر في الكتالوج العام إلا «المتوفرة».
> العميل يفتح صفحة السيرة من الكتالوج ويحجز بإدخال رقم جواله؛ يصل الطلب للمكتب في صفحة العمالة.
> كل مكتب له **رابط كتالوج خاص** يرسله لعملائه؛ ويظهر الرابط أيضاً في لوحة مدير المنصة بجانب كل مكتب.

## ملاحظات للإنتاج
- كلمات المرور مخزّنة بتجزئة scrypt مع salt.
- غيّر `JWT_SECRET` وكلمات المرور الافتراضية قبل النشر.
- قاعدة MySQL تضمن بقاء البيانات؛ للمرفقات يُفضّل التوسّع لاحقاً لخدمة تخزين (S3) بروابط موقّعة.
- فعّل HTTPS وسجل التدقيق، والتزم بنظام حماية البيانات الشخصية (PDPL).

## الاشتراكات والدفع (Tap)
- الخطط الافتراضية: شهري 200، سنوي 800، مدى الحياة 2000 ريال (+15% ضريبة). فترة تجريبية 14 يوماً تلقائياً لكل مكتب جديد.
- عند انتهاء الاشتراك/التجربة تُقفل أدوات المكتب (ترجع 402 code=subscription_required)، وتبقى صفحة `subscription.html` متاحة.
- مدير المنصة يدير الاشتراكات من `admin.html` (منح تجربة / تفعيل خطة يدوياً) — التفعيل اليدوي مفيد للتحويل البنكي.

### متغيّرات البيئة الإضافية
- `SUB_TRIAL_DAYS` (افتراضي 14): مدة الفترة التجريبية.
- `TAP_SECRET_KEY`: مفتاح Tap السري (sk_test_... / sk_live_...). بدونه يعمل النظام بالتفعيل اليدوي فقط.
- `APP_URL`: رابط الموقع العام (مثل https://smartistiqdam.com) لروابط العودة من Tap.

### تدفّق الدفع عبر Tap (المرحلة 2)
1. المكتب يختار خطة في `subscription.html` ويضغط دفع.
2. الخادم ينشئ Charge في Tap ويحوّل المستخدم لصفحة الدفع.
3. بعد الدفع يعود إلى `subscription.html?tap_id=...` فيتحقق الخادم عبر `/api/subscription/verify` ويُمدّد الاشتراك ويسجّل الدفعة.
4. يوجد أيضاً Webhook احتياطي على `/api/subscription/tap-webhook`.
